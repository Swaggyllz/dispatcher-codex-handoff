AI Project Operating System

超级模式：把 AI 从聊天工具变成项目工程代理

一套更适合真实项目的 AI 工作方法：先锁边界，再拆任务，再执行验证，最后留下可交接的证据。

Multi-agent PM 目标循环 证据账本 风险门禁 文档沉淀

很多人用 AI 做项目，其实还停留在“聊天”：问一句、答一句、改一段、报错再问。 AI 看起来很忙，但项目状态并没有真正变清楚。超级模式解决的不是“让 AI 更会说”， 而是让 AI 按工程项目的方式持续交付。

普通 AI 工作流的问题

大部分人用 AI 做项目，流程大概是这样：

我提需求
-> AI 给方案
-> 我让它写代码
-> 报错
-> 我复制报错
-> AI 再修
-> 最后不知道到底完成没有

这个流程最大的问题是：没有边界，没有验收，没有证据，没有交接。AI 可能做了很多动作， 但项目仍然是模糊的。

超级模式是什么

超级模式不是一个“神奇 prompt”。它更像一个项目操作系统。启动后，AI 不只是回答问题， 而是按一套项目治理流程推进。 

先读项目事实源
-> 锁定范围和边界
-> 补齐目标和验收标准
-> 用 PM 思维拆任务
-> 必要时分多个 agent 研究
-> 小步实现
-> 真实验证
-> 失败就诊断修复
-> 沉淀文档
-> 最后给可交接汇报
普通 loop 只是让 AI 一直干。超级模式是让 AI 在一个有边界、有证据、有交接机制的项目系统里干。

第一步：先读事实，不要先写代码

复杂项目最怕凭记忆推进。超级模式启动后，第一步必须读取项目事实源。

PROJECT_MANUAL.md
README
git status
最新 commit
最近验证结果
当前服务状态
Scope Boundaries
Release Strategy

这一步看起来慢，但能避免非常多返工：比如改错仓库、覆盖旧版本、重复实现、忘记上一次已经验证过的内容。

第二步：补齐目标和验收标准

很多项目不是 AI 做不出来，而是“完成”的定义不清楚。

目标是什么
不做什么
哪些文件可以改
哪些文件不能动
什么叫完成
要跑哪些验证
是否需要更新文档
是否允许 commit / push / 发布

如果这些没写清楚，超级模式会先从代码、文档、配置、日志里推断一版合理验收标准， 然后再执行，而不是上来就空想方案。

第三步：用 Multi-agent PM 拆任务

复杂项目不要让一个 AI 从头想到尾。主 AI 先当 PM，把任务拆成独立问题，再派不同 agent 去查。

PM Agent

拆目标、定验收、排优先级，负责让项目不跑偏。

Explorer Agent

只查事实，不改文件，输出缺口、证据和风险。

Worker Agent

按限定范围实现，不碰无关文件，不做无关重构。

Reviewer / Verifier

找风险、跑验证、复现问题，用命令结果说话。

多 agent 的价值不是热闹，而是减少盲区。每个 agent 都要有明确任务卡：查什么、允许看什么、禁止做什么、输出什么。

第四步：目标循环执行

真正实现时，超级模式按目标循环推进。

读取现状
-> 判断问题
-> 制定小计划
-> 小步实现
-> 运行验证
-> 失败诊断
-> 修复
-> 再验证

这里最重要的一条是：不要停在方案。能操作就操作，能验证就验证，验证失败就继续诊断。 不说“你可以试试运行测试”，而是直接运行测试。

第五步：验证必须是真实证据

超级模式里不能说“应该没问题”。必须有验证命令和结果。

L1 单点测试
L2 模块测试
L3 workspace 测试
L4 lint / typecheck / build
L5 服务启动验证
L6 发布 readiness
L7 浏览器或人工验收

最终汇报必须写清楚：跑了什么命令、结果是什么、是否通过、失败过什么、怎么修的、还有什么风险。 没有证据的“完成”，都不算完成。

第六步：风险门禁

超级模式不是无限自动化。有些事 AI 可以直接做，有些事必须停下来确认。

可以直接做：
读文件、查日志、跑测试、小范围改代码、更新文档、启动本地服务

必须确认：
删除文件、清库、数据库迁移、生产操作、发布 release、git push、
覆盖旧仓库、修改账号/权限/密钥、不可逆操作

真正可用的自动化，不是什么都敢干，而是知道什么时候必须停。

第七步：沉淀文档，让项目可接力

超级模式每完成一个阶段，都要更新项目状态。项目不能靠聊天记录活着，要靠文件系统活着。

PROJECT_MANUAL.md：项目事实源
WORK_ORDERS.md：任务状态
DECISIONS.md：关键决策
EVIDENCE.md：验证证据
RISKS.md：风险台账
HANDOFF.md：下一窗口接手摘要

这样第二天换窗口、换 agent、换人接手，都能知道项目做到哪，哪些东西验证过，哪些边界不能碰。

超级模式启动词

完整启动词可以这样写：

启动超级模式来做这个项目。

先读项目手册、README、git 状态、最新提交和验证记录。
锁定 Scope 边界和发布策略。
补齐目标、范围、验收标准和禁止事项。
必要时用 Multi-agent PM 拆任务、分 agent、交叉核验。
按目标循环执行：读现状 -> 判断问题 -> 小计划 -> 实现 -> 验证 -> 修复 -> 再验证。
关键决策写记录，验证结果留证据，风险写清楚。
最后更新项目手册和交接摘要，给高信号汇报。
不要停在方案；能操作就操作；发布、删除、覆盖、迁移、push 前必须确认。

短版本就是：

启动超级模式，按项目操作系统推进。

最后的理解

超级模式真正有用的地方，不是 prompt 写得多漂亮，而是它把 AI 从“回答问题的人”， 变成了“推进项目的工程代理”。 

尊重事实
遵守边界
持续执行
留下证据

这四件事做好，AI 才能真的参与复杂项目。否则它只是一个很会说话的工具。 超级模式不是让项目变复杂，而是让复杂项目不失控。
