# Dispatcher v0.2.1 Codex Handoff Follow-up

Release date: 2026-06-18

Dispatcher v0.2.1 publishes the post-v0.2.0 Dispatcher 2.0 follow-up work to
the Dispatcher 2.0 repository:
[Swaggyllz/dispatcher-codex-handoff](https://github.com/Swaggyllz/dispatcher-codex-handoff).

This release must not be pushed to the existing Dispatcher 1.0 repository.

## Summary

v0.2.1 hardens the Codex Handoff Router after the initial v0.2.0 emergency
handoff release. It adds planned handoff creation from reliable upstream quota
snapshots, keeps background fallback continuation explicitly opt-in, persists
tagged streaming continuation outcomes, and improves the dashboard workflow for
reviewing fallback output before returning to the primary Codex route.

## Included

- Planned handoff packages from reliable `x-ratelimit-limit-*` and
  `x-ratelimit-remaining-*` header pairs when normalized headroom is at or
  below `DISPATCHER_PLANNED_HANDOFF_THRESHOLD`.
- Default planned handoff threshold remains `0.10`, but no planned handoff is
  created without reliable upstream header pairs.
- Background fallback continuation through `provider-auto`, disabled by default
  and enabled only when `DISPATCHER_HANDOFF_AUTO_CONTINUE=1`.
- Tagged non-streaming and streaming provider-auto continuations persisted as
  terminal success or terminal failure telemetry.
- `/v1/telemetry` surfaces latest quota snapshot, latest handoff package, and
  latest handoff continuation state.
- Simple dashboard displays saved fallback continuation source/status, copyable
  review prompt, and explicit primary Codex review action.
- Release metadata aligned to `0.2.1`.

## Not Included

- Pushing or replacing the Dispatcher 1.0 repository.
- Claude Code-specific handoff behavior.
- Hosted Responses tool emulation for third-party providers.
- Hidden reasoning or private context migration.
- Claims that fallback models are equivalent to Codex-native execution.
- Automatic fallback execution unless the operator explicitly enables
  `DISPATCHER_HANDOFF_AUTO_CONTINUE=1`.
